# ZMK Lily58 Keyboard Configuration Agent

You assist with ZMK firmware configuration for this Lily58 split keyboard repository.

## Scope

- Keymap and behavior changes in `config/lily58.keymap`
- Kconfig changes in `config/lily58.conf`
- Build matrix and version pinning in `build.yaml`, `config/west.yml`, `.github/workflows/build.yml`

## Repository Baseline

- Keyboard: Lily58 split wireless
- Board: nice_nano (v0.3.0 pinned setup)
- Display: nice_view_adapter + nice_view
- Build pinning:
  - workflow uses `zmkfirmware/zmk/.github/workflows/build-user-config.yml@v0.3.0`
  - west manifest uses `revision: v0.3.0`

## Hard Rules (From Real CI Failures)

1. Do not add `sensor = <...>;` inside `compatible = "zmk,behavior-sensor-rotate-var"` behavior nodes.
2. Behavior labels must be unique (no duplicates in `behaviors { ... }`).
3. `sensor-bindings` entries must match the number of sensors available for the build target.
4. Do not enable `CONFIG_ZMK_BACKLIGHT=y` unless DTS provides `chosen { zmk,backlight = ...; }`.
5. For this repo, prefer encoder mappings via `&kp` keycodes unless a verified backlight node exists.

## Typical Safe Patterns

- Define one reusable rotate-var behavior:
  - `bindings = <&kp>, <&kp>;`
- Base layer sensor bindings:
  - `sensor-bindings = <&rot_kp C_VOL_UP C_VOL_DN>;`
- Lower layer sensor bindings (host brightness fallback):
  - `sensor-bindings = <&rot_kp C_BRI_UP C_BRI_DN>;`

## Plan Estable (Solo Lo Validado)

Objetivo: mantener una base estable y repetible para uso diario.

1. Mantener versionado fijo en v0.3.0
  - `config/west.yml`: `revision: v0.3.0`
  - `.github/workflows/build.yml`: `@v0.3.0`
2. Mantener board y shields conocidos
  - `build.yaml`: `board: nice_nano`
  - Shields: `lily58_left nice_view_adapter nice_view` y `lily58_right nice_view_adapter nice_view`
3. Mantener features activas que ya funcionan
  - `CONFIG_ZMK_BLE=y`
  - `CONFIG_ZMK_DISPLAY=y`
  - `CONFIG_EC11=y`
  - `CONFIG_ZMK_STUDIO=y`
4. Mantener desactivado lo no soportado por DTS actual
  - `CONFIG_ZMK_BACKLIGHT=n` (hasta tener `zmk,backlight` en `chosen`)
5. Mantener encoder con comportamiento único y simple
  - Un solo behavior `rot_kp` con `bindings = <&kp>, <&kp>;`
  - Un `sensor-bindings` por layer
6. Flujo de cambios seguro
  - Cambiar una cosa por vez
  - Correr CI
  - Si falla, tomar el primer `error:` antes de `FATAL ERROR`

## Fuera de Este Plan (Hasta Nueva Validacion)

- Rehabilitar backlight en Kconfig sin nodo DTS correspondiente
- Agregar multiples `sensor-bindings` por layer sin confirmar numero de sensores
- Cambiar board naming fuera de `nice_nano` dentro de este pin de version

## Build/CI Troubleshooting Order

1. Check first real error before `FATAL ERROR`.
2. If `gen_defines.py failed`, inspect DTS properties/behavior nodes.
3. If `keymap.c` shows excess initializer elements, reduce `sensor-bindings` count.
4. If `backlight.c` static assert fails, disable `CONFIG_ZMK_BACKLIGHT` or add proper DTS chosen node.
